**作者**：产品
**适用系统**：内部问卷系统（hr.mmcb.top）
**关联文档**：《内部问卷系统 · 项目说明文档》

## 一、背景与目标

### 1.1 背景

现有内部问卷系统支持管理员创建问卷、配置白名单、定向触达指定人员填写。360 环评是一种**特殊的评价型问卷场景**：在某个组织（如技术组）内部，成员之间需要相互评价（他评），同时对自己评价（自评），从而形成多维度的人才评估数据。

与普通问卷"一份问卷发给一批人、每人填一次"不同，360 环评的本质是 **"评价关系"驱动**：同一个人需要对组内每位同事各填一份他评问卷，外加一份自评问卷。普通员工走统一规则，而**领导人员**因评价范围、问卷内容不同，需要单独配置（白名单机制）。

### 1.2 目标

1. 在不推翻现有问卷/白名单/联系人体系的前提下，**最大化复用现有能力**，增量实现 360 环评。
2. 普通员工的评价关系由系统**按组织架构自动生成**，零维护。
3. 领导人员通过**白名单单独配置**，例外集中处理。
4. 提供一张**复核列表（覆盖度看板）**，让 HR 一眼看出谁未配置自评 / 他评，便于人工兜底。
5. 自动生成时若发现异常（如单人组无法互评），**主动提示人工处理**，不静默跳过。

### 1.3 本期明确的产品决策（已与需求方确认）

| 议题 | 决策 |
|------|------|
| 互评人数上限 / 抽样 | **不做**抽样和限额，组内逐人互评 |
| 单人组无法互评 | 视为异常，走**白名单人工特殊配置**；生成时主动提示 |
| 领导角色绑定维度 | 领导人数少，**白名单仅按人绑定**即可，不需绑组织节点 |
| 匿名性 | 他评**结果对被评人不可见**（仅管理员可见），无需匿名打散逻辑 |
| 批次内人员变动 | 评价关系表支持**人工增删配置**，不做自动重算 |

## 二、核心概念与术语

| 术语 | 含义 |
|------|------|
| 评价批次（Cycle） | 一次完整的 360 环评活动，圈定参与范围、绑定问卷模板、生成评价关系 |
| 评价关系（Relation） | 一条"谁评谁、用哪份问卷"的记录，是整个功能的最小单元 |
| 评价人（Rater） | 填写评价的人 |
| 被评人（Ratee） | 被评价的对象 |
| 关系类型（RelationType） | self（自评）/ peer（同级他评）/ leader（领导评价） |
| 问卷模板 | 自评、他评、领导评价三类问卷，复用现有 Survey |
| 领导白名单 | 标记为领导、需单独配置评价关系的人员清单 |
| 复核列表 | 全员视角的配置覆盖度看板，用于人工复核遗漏 |

---

## 三、与现有系统的复用关系（核心）

设计的关键原则是：**能复用的绝不新建**。下表是 360 环评每个能力点到现有系统的映射。

| 360 能力 | 复用现有 | 复用方式 / 改造点 |
|----------|----------|-------------------|
| 问卷模板（自评/他评/领导评价） | **Survey** | 三类问卷即三份 Survey；可沿用 `assessment` 类型或新增枚举 `evaluation`。题目编辑器、题型、联动逻辑全部复用 |
| 组织架构 / 人员名单 | **Contact** | `department` 字段作为"组织/组"分组依据，`position` 作为职位，`tags` 可打"领导"标记 |
| 领导白名单 | **Whitelist 思路** | 复用白名单"按人圈选"的交互范式，但语义升级为"评价关系配置"，落到新表 `eval_relation` |
| 填写身份认证 | **企微 OAuth + fill-auth.guard** | 完全复用，员工扫码后识别身份 |
| 答卷存储 | **SurveyResponse** | 扩展一个 `ratee_contact_id` 字段，放开"每人每问卷限填一次"为"每人对每个被评人限填一次" |
| 管理员评分/评语 | **ResponseComment** | 直接复用 |
| 数据导出 | **CSV 导出** | 复用，导出时增加"被评人"列 |
| 后台框架 / 鉴权 / 文件夹 | **NestJS / JWT / SurveyFolder** | 评价批次可归入文件夹体系管理 |

**结论**：360 环评新增的核心只有两张表——**评价批次（eval_cycle）** 和 **评价关系（eval_relation）**，再加上 `SurveyResponse` 的一个字段扩展。其余全部复用，开发量可控。

## 四、数据模型设计

### 4.1 新增表：评价批次 eval_cycle

```prisma
model EvalCycle {
  id              Int       @id @default(autoincrement()) @db.UnsignedInt
  name            String    @db.VarChar(200)              // 批次名称，如"2026 Q2 技术部 360 环评"
  scopeDepartment String?   @map("scope_department")      // 参评范围（按部门/组），为空表示自定义人员
  selfSurveyId    Int?      @map("self_survey_id")         // 自评问卷模板
  peerSurveyId    Int?      @map("peer_survey_id")         // 他评问卷模板
  leaderSurveyId  Int?      @map("leader_survey_id")       // 领导评价问卷模板
  status          String    @default("draft")             // draft/generating/published/closed
  createdBy       Int       @map("created_by") @db.UnsignedInt
  createdAt       DateTime  @default(now()) @map("created_at")
  updatedAt       DateTime  @updatedAt @map("updated_at")

  relations       EvalRelation[]
  @@map("eval_cycles")
}
```

### 4.2 新增表：评价关系 eval_relation（功能基石）

```prisma
model EvalRelation {
  id            Int      @id @default(autoincrement()) @db.UnsignedInt
  cycleId       Int      @map("cycle_id") @db.UnsignedInt
  raterContactId Int     @map("rater_contact_id") @db.UnsignedInt  // 评价人
  rateeContactId Int     @map("ratee_contact_id") @db.UnsignedInt  // 被评人
  relationType  String   @map("relation_type")    // self / peer / leader
  surveyId      Int      @map("survey_id") @db.UnsignedInt          // 该关系使用的问卷模板
  source        String   @default("auto")          // auto（规则生成）/ manual（人工配置）
  status        String   @default("pending")        // pending / submitted
  responseId    Int?     @map("response_id")         // 提交后回填，关联 SurveyResponse
  createdAt     DateTime @default(now()) @map("created_at")

  cycle  EvalCycle @relation(fields: [cycleId], references: [id])
  @@unique([cycleId, raterContactId, rateeContactId], map: "uk_cycle_rater_ratee")
  @@map("eval_relations")
}
```

> 自评关系即 `raterContactId == rateeContactId` 且 `relationType = self`，无需特殊表结构。

### 4.3 扩展表：SurveyResponse

```prisma
model SurveyResponse {
  // ...现有字段...
  rateeContactId Int?  @map("ratee_contact_id")   // 新增：被评人（普通问卷为空）
  // 原"每人每问卷限填一次"约束 → 调整为按 (survey, rater, ratee) 唯一
}
```

### 4.4 领导白名单的落地方式

不新建独立表，采用**轻量标记 + 关系表**组合：

- **识别领导**：在 `Contact.tags` 打"领导"标签（或批次内单独勾选一份领导名单）。
- **配置领导关系**：领导的所有 `eval_relation` 由管理员在批次内**手动配置**（source = manual），自动生成时跳过领导作为被评人/评价人的默认规则。

这样领导白名单本质就是"哪些 contact 走人工配置"，无需额外表，符合"领导少、仅按人绑定"的决策。

## 五、评价关系生成规则

### 5.1 自动生成（普通员工）

输入：评价批次 + 参评范围（某部门/组的 Contact 集合）+ 领导白名单。

对范围内每位**普通员工** P（不在领导白名单内）：

1. **自评**：生成一条 `self` 关系，rater = ratee = P，使用自评模板。
2. **同级他评**：对同组其他每位**普通员工** Q，生成一条 `peer` 关系，rater = P，ratee = Q，使用他评模板。
3. 领导默认**不进入**自动互评网（既不评、也不被普通规则评），全部交由白名单人工配置。

分组依据：`Contact.department` 字段值相同视为同组。

### 5.2 人工配置（领导 + 异常）

以下情况走管理员手动配置 `eval_relation`（source = manual）：

- **领导人员**：评价范围、问卷模板可能不同（如"全组评领导、领导评下属"），由管理员逐条配置。
- **单人组**：组内只有 1 人时无法互评，自动生成阶段标记异常，由管理员决定（跨组评、豁免他评、或仅保留自评）。
- **批次内人员变动**：新入职/离职/借调，管理员在关系表中手动增删对应关系。

### 5.3 异常主动提示（关键体验）

自动生成执行后，系统输出一份**生成报告**，对以下情况高亮提示、要求人工确认：

- 单人组成员（无可互评对象）
- 在领导白名单但尚未配置任何关系的人员
- 部门字段为空、无法归组的人员
- 范围内既不在自动结果、又不在人工配置中的"双漏"人员

> 设计要点：生成不是"跑完即结束"，而是"跑完→给出待人工处理清单"，把规则覆盖不到的部分显式交还给人。

## 六、功能模块设计

### 6.1 评价批次管理（管理员端）

- 新建批次：填写名称、选择参评范围（部门/组或自定义人员）、绑定三类问卷模板。
- 批次状态：草稿 → 生成关系 → 发布 → 关闭。
- 批次列表：展示参评人数、关系数、配置完整度、状态。

### 6.2 问卷模板配置

直接复用现有问卷编辑器，创建三份问卷（自评 / 他评 / 领导评价），在批次中分别绑定。题型、必填、联动等能力全部沿用。

### 6.3 领导白名单配置

- 在批次内从参评人员中勾选"领导"，加入领导白名单。
- 为每位领导配置评价关系：谁评他（被评人侧）、他评谁（评价人侧）、用哪份问卷。
- 交互范式复用现有白名单"搜索联系人→勾选添加"。

### 6.4 一键生成评价关系

- 按 5.1 规则批量生成普通员工关系。
- 生成后弹出**生成报告 + 异常清单**（见 5.3），管理员逐项处理。
- 支持重新生成（覆盖 source=auto 的记录，保留 source=manual 的人工配置不被冲掉）。

### 6.5 复核列表（覆盖度看板）★

按**被评人维度**展开全员，核心是把"异常态"高亮，让 HR 快速定位遗漏。

字段示例：

| 姓名 | 部门/组 | 角色 | 自评 | 被几人评（他评·被评侧） | 评几人（他评·评价侧） | 配置来源 | 状态 |
|------|---------|------|------|------------------------|----------------------|----------|------|
| 张三 | 技术组 | 普通员工 | ✅ | 4 | 4 | 自动 | 完整 |
| 李四 | 技术组 | 领导 | ⚠️ 未配 | 0 | 0 | 白名单·待配 | **异常** |
| 王五 | （空） | 普通员工 | ❌ | 0 | 0 | 未覆盖 | **双漏** |

功能点：

- 顶部统计：参评总数 / 已完整配置数 / 异常数。
- 筛选："仅看异常"、按部门、按角色、按配置来源。
- 异常类型标识：未配自评、无人评价、领导待配、未归组、双漏。
- 点击行可跳转到该人的关系配置进行补配。

### 6.6 填写端（员工端）

- 员工企微扫码进入后，看到**"待我填写"列表**：1 份自评 + 对 N 位同事的 N 份他评。
- 每份他评清晰标注**被评人姓名**，逐份填写、逐份提交。
- 提交后该关系状态置为 submitted，列表显示已完成进度（如 3/5）。
- 复用现有填写鉴权：非参评人/无关系的人访问提示无权限。

### 6.7 结果查看与导出

- 管理员按被评人聚合查看其收到的所有评价（自评 + 各他评 + 领导评价）。
- **他评结果对被评人本人不可见**，仅管理员可见。
- CSV 导出复用现有能力，增加"被评人"列，便于线下分析。

## 七、关键流程

### 7.1 管理员配置流程

```
创建问卷模板（自评/他评/领导评价）
      ↓
新建评价批次，选择参评范围 + 绑定三份模板
      ↓
勾选领导白名单
      ↓
一键生成普通员工评价关系
      ↓
查看生成报告 + 异常清单 → 人工补配（领导/单人组/变动人员）
      ↓
复核列表确认全员"完整" → 发布批次
```

### 7.2 员工填写流程

```
企微扫码进入 → 看到"待我填写"列表（自评 + N 份他评）
      ↓
逐份填写并提交 → 进度更新
      ↓
全部完成 → 显示完成页
```

## 八、边界与异常处理

| 场景 | 处理方式 |
|------|----------|
| 单人组无法互评 | 自动生成标记异常，提示人工配置（跨组/豁免/仅自评） |
| 部门字段为空 | 无法归组，列入异常清单，人工指定 |
| 领导未配置关系 | 复核列表标"领导·待配"异常 |
| 既未自动覆盖又未人工配置（双漏） | 复核列表最高优先级红色提示 |
| 批次中途人员变动 | 管理员在关系表手动增删，不自动重算 |
| 重复生成 | 仅覆盖 source=auto 记录，人工配置（manual）保留 |
| 同一人跨部门 | 本期以 Contact.department 单值归组；如需多组参评，人工补配 |

## 九、方案选型与推荐

### 9.1 三种关系生成方案对比

| 方案 | 说明 | 优点 | 缺点 |
|------|------|------|------|
| A 全手动 | 管理员逐人配置所有关系 | 极灵活 | 工作量大、易遗漏，复核列表沦为救火工具 |
| B 全自动 | 完全靠规则生成，无例外通道 | 快、零遗漏 | 领导/单人组等例外塞不进规则，硬塞污染逻辑 |
| **C 规则自动 + 白名单例外（推荐）** | 普通员工自动生成，领导/异常走人工 | 80/20 最优、规则与例外解耦、复核列表聚焦例外 | 需维护白名单与异常处理入口 |

### 9.2 推荐方案 C 的原因

1. **80/20 最优**：绝大多数普通员工自动生成、零维护；人力集中在真正需要判断的领导与异常上。
2. **规则与例外解耦**：白名单是独立开关，不污染自动规则；未来扩展"跨部门评价""上级评价"互不干扰。
3. **复核列表真正发挥兜底价值**：大部分自动生成，复核只需聚焦"例外是否配好"，契合"人工复核谁未设置"的核心诉求。
4. **复用成本最低**：自动规则 + 关系表是增量逻辑，问卷、白名单交互、填写鉴权、导出全部复用现有系统。

## 十一、开发改造点清单（给研发参考）

1. **数据库**：新增 `eval_cycles`、`eval_relations` 两张表；`survey_responses` 增加 `ratee_contact_id`，调整唯一约束。
2. **后端**：批次 CRUD、关系自动生成器（按 department 分组）、生成报告/异常计算、复核列表聚合查询、填写端"待我填写"接口、提交回填关系状态。
3. **前端**：批次管理页、领导白名单配置、生成报告弹窗、复核列表（含异常筛选）、员工端待填写列表 + 多份他评填写。
4. **复用项（不改造或小改）**：问卷编辑器、企微 OAuth、ResponseComment 评分评语、CSV 导出（增列）、文件夹归类。
